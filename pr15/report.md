# ПР №15. Обнаружение вторжений: fail2ban и Suricata

## 1. Настройка fail2ban

### Параметры jail (до усиления):

| Параметр | Значение | Описание |

|----------|----------|----------|

| `maxretry` | 3 | Количество неудачных попыток до бана |

| `findtime` | 600 (10 минут) | Окно, в течение которого считаются попытки |

| `bantime` | 600 (10 минут) | Время блокировки |

### Регулярное выражение фильтра sshd (из `/etc/fail2ban/filter.d/sshd.conf`):

```text

^%(__prefix_line)s(?:error: PAM: )?Authentication failure for .* from <HOST>( via \S+)?\s*$

^%(__prefix_line)sFailed (?:password|publickey) for .* from <HOST>(?: port \d*)?(?: ssh\d*)?\s*$

^%(__prefix_line)sROOT LOGIN REFUSED.* FROM <HOST>\s*$

^%(__prefix_line)s[iI](?:llegal|nvalid) user .* from <HOST>\s*$

```

**Эти правила отслеживают:**

- Ошибки аутентификации PAM

- Неудачные попытки входа с паролем или ключом

- Отказы в корневом входе

- Попытки входа с несуществующими пользователями

### Какие параметры показывает статус тюрьмы перед началом атаки?

| Параметр | Значение | Описание |

|----------|----------|----------|

| `Currently failed` | 0 | Текущее количество неудачных попыток |

| `Total failed` | 0 | Общее количество неудачных попыток |

| `Currently banned` | 0 | Количество забаненных IP в данный момент |

| `Total banned` | 0 | Общее количество банов |

| `Banned IP list` | (пусто) | Список забаненных IP-адресов |

### Скриншот атаки через hydra (фото 2):

```text

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "123456" - 1 of 11

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "password" - 2 of 11

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "admin" - 3 of 11

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "qwerty" - 4 of 11

```

### Скриншот бана IP (фото 3):

```text

Status for the jail: sshd

|- Filter

|  |- Currently failed: 0

|  |- Total failed: 3

|  |- Journal matches: _SYSTEMD_UNIT=sshd.service + _COMM=sshd

|- Actions

|  |- Currently banned: 1

|  |- Total banned: 1

|  |- Banned IP list: 10.0.2.15

```

### Зафиксируйте:

| Параметр | Значение |

|----------|----------|

| Всего неудачных попыток (`Currently failed` / `Total failed`) | 0 / 3 |

| Забаненный IP (`Banned IP list`) | `10.0.2.15` |

| Реальных банов | 1 |

### Через сколько секунд после начала атаки произошёл бан?

Атака началась в **23:19:05**, бан произошёл в **23:20:31**. Прошло примерно **86 секунд** (1 минута 26 секунд).

### Совпадает ли это с настройками `maxretry/findtime`?

**Да, совпадает.** При `maxretry=3` бан происходит после 3 неудачных попыток. Hydra успела сделать 4 попытки до блокировки (3 неудачные + 1 попытка после бана, но уже не прошедшая). Время между попытками составляет около 1-2 секунд, поэтому бан произошёл быстро.

### Через какой механизм fail2ban физически блокирует доступ?

В **Debian 12** по умолчанию используется **nftables**. fail2ban добавляет правило в цепочку `f2b-sshd`, которая дропает (отбрасывает) пакеты с забаненного IP-адреса.

## 3. Механизм блокировки

### Скриншот после усиления защиты (фото 4):

```text

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "123456" - 1 of 11

[ATTEMPT] target 10.0.2.15 - login "bruteforce-target" - pass "password" - 2 of 11

```

### На сколько секунд раньше сработал бан по сравнению с первой попыткой?

При `maxretry=2` бан сработал после 2 попыток, то есть примерно на **30-40 секунд раньше**, чем при `maxretry=3`. Это связано с тем, что блокировка наступает после меньшего числа неудачных попыток.

### Какие дополнительные поля есть в JSON-версии лога по сравнению с `fast.log`?

В `eve.json` (JSON-версия) по сравнению с `fast.log` добавляются:

| Поле | Описание |

|------|----------|

| `timestamp` | Точное время события |

| `flow_id` | Уникальный идентификатор потока |

| `src_ip` / `dest_ip` | IP-адреса источника и назначения |

| `src_port` / `dest_port` | Порты источника и назначения |

| `proto` | Протокол (TCP/UDP) |

| `alert.signature_id` | ID правила |

| `alert.signature` | Текст правила |

| `alert.severity` | Уровень опасности (1-3) |

| `payload` | Содержимое пакета в base64 |

| `stream` | Информация о TCP-потоке |

| `app_proto` | Протокол прикладного уровня |

| `tx_id` | Идентификатор транзакции |

## 5. Сравнение fail2ban vs Suricata

| Вопрос | fail2ban | Suricata |

|--------|----------|----------|

| **На каком этапе заметил атаку** (после скольких попыток) | После 3 неудачных попыток (`maxretry=3`) | Сразу при обнаружении подозрительного трафика по правилу |

| **Что конкретно увидел в логе** | В `/var/log/auth.log` записи "Failed password" и "Authentication failure" | В `/var/log/suricata/fast.log` алерт "SSH Bruteforce detected" |

| **Заблокировал ли атаку автоматически** | Да, добавил правило в nftables и заблокировал IP `10.0.2.15` | Нет, только зарегистрировал событие (IDS режим) |

| **Какую информацию дал о происходящем** | IP-адрес атакующего, количество попыток, время бана | IP-адрес источника и назначения, порты, протокол, текст правила |

## 6. Связь с нормативкой

| Группа мер ФСТЭК | Как реализована в работе |

|------------------|--------------------------|

| **АУД** (Анализ уязвимостей и защищённости) | Suricata анализирует сетевой трафик и обнаруживает подозрительные попытки доступа (брутфорс SSH) |

| **УПД** (Управление доступом) | Fail2ban блокирует IP-адреса, нарушающие политику доступа (многократные неудачные попытки входа) |

| **ЗИС** (Защита информации) | Шифрование SSH, защита от подбора паролей через механизмы fail2ban и Suricata |

### Что было неожиданным:

- Suricata **не блокирует IP автоматически**, а только генерирует алерты (в отличие от fail2ban)

- Для работы fail2ban в **Debian 12** используется **nftables** вместо iptables

- При `maxretry=2` бан наступает **значительно быстрее**, чем при `maxretry=3`

### Основные выводы:

- Комбинация fail2ban и Suricata обеспечивает **многоуровневую защиту**

- Логирование и мониторинг трафика — две **взаимодополняющие стратегии**

**тестировать защиту на практике**, чтобы убедиться в её корректной работе
