**Можно ли прочитать исходные данные из зашифрованного файла?**  

Нет, потому что файл зашифрован асимметричным шифрованием. Без приватного ключа получателя невозможно расшифровать данные. Содержимое выглядит как случайный набор символов.

**Что произойдёт если ввести неверный пароль при расшифровке?**  

GPG выдаст ошибку: `gpg: decryption failed: Bad session key` или `gpg: decryption failed: No secret key`. Расшифровка не произойдёт, и вы не получите исходные данные.

**Почему нельзя расшифровать файл зашифрованный для другого пользователя?**  

Потому что файл зашифрован с использованием **публичного ключа** получателя. Расшифровать его можно только с помощью **соответствующего приватного ключа**, который есть только у получателя. Даже если у вас есть публичный ключ этого пользователя, вы не сможете расшифровать файл — для этого нужен его приватный ключ.

## 3. Цифровая подпись

**Вывод `gpg --verify` (Good signature):**

gpg: Signature made Wed 17 Jun 2026 04:00:51 PM +05

gpg: using RSA key F57F1FE60363A99EAE3977CF59A501BD6C3E3FAD

gpg: Good signature from "Ivane mare.f@bk.ru" [ultimate]

**Что именно проверяет эта команда?**  

Команда `gpg --verify` проверяет:

1. **Целостность** — не изменилось ли содержимое файла после подписания

2. **Подлинность** — действительно ли файл подписан владельцем указанного ключа

3. **Доверие** — проверяет цепочку доверия к ключу (на основе `trustdb`)

---

**Вывод `gpg --verify` после изменения файла (BAD signature):**

gpg: Signature made Wed 17 Jun 2026 04:00:51 PM +05

gpg: using RSA key F57F1FE60363A99EAE3977CF59A501BD6C3E3FAD

gpg: BAD signature from "Ivane mare.f@bk.ru" [ultimate]

**Что это означает и почему подпись перестала быть действительной?**  

Подпись создаётся на основе **хеша (дайджеста)** содержимого файла. При изменении даже одного символа хеш меняется, и при проверке подписи вычисленный хеш не совпадает с тем, что был подписан. Поэтому GPG сообщает о недействительности подписи (`BAD signature`).

**Зачем проверять подпись локально если GitHub уже показывает Verified?**

- GitHub может быть скомпрометирован или показывать неверную информацию

- Вы можете работать в офлайн-режиме без доступа к GitHub

- Доверять своим инструментам (локальный GPG) безопаснее, чем полагаться на сторонний сервис

- Локальная
проверка гарантирует, что подпись была создана именно вашим ключом

## 5. rm vs shred

**Что увидели на этот раз?**  

При использовании `rm` файл исчезает из списка, но данные остаются на диске до момента перезаписи. При использовании `shred` содержимое файла многократно перезаписывается случайными данными, после чего файл удаляется. Восстановить данные после `shred` практически невозможно.

**Чем `shred` принципиально отличается от `rm` на уровне работы с блоками диска?**

| Операция | Что происходит с данными на диске | Можно восстановить? |

|----------|----------------------------------|---------------------|

| `rm` | Удаляется только запись в метаданных (inode). Блоки с данными помечаются как свободные, но сами данные остаются на диске | Да, пока блоки не были перезаписаны новыми данными. Восстановление возможно с помощью специальных утилит (testdisk, photorec) |

| `shred` | Содержимое блоков многократно перезаписывается случайными или заранее заданными данными (по умолчанию 3 прохода: 0xFF, 0x00, случайные) | Нет (практически). После перезаписи восстановить исходные данные невозможно |

**Сценарий когда нужно затирать свободное место:**

- При продаже или передаче диска/компьютера другому человеку

- При утилизации старого оборудования

- При работе с конфиденциальными данными (банковская информация, персональные данные, коммерческая тайна)

- В системах с требованиями к безопасности (ГОСТ, PCI DSS, GDPR)

- После удаления файлов с паролями или ключами шифрования
