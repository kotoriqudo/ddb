# ПР №10. Лицензирование и open-source compliance

## 1. Выбор лицензии

**Выбранная лицензия:** MIT  

**Тип:** разрешительная (permissive)

### Что разрешает:

- Использование кода в любых целях, включая коммерческие

- Модификацию и адаптацию под свои нужды

- Распространение в исходном или скомпилированном виде

- Сублицензирование и включение в проприетарные проекты

### Что требует от пользователей кода:

- Сохранение уведомления об авторских правах (Copyright notice) во всех копиях

- Сохранение текста лицензии MIT

- Указание оригинального автора

### Может ли кто-то продавать ваш код:

**Да**, могут, даже без вашего разрешения.  

Единственное условие — сохранить ваше авторство в коде (Copyright notice).

**Вывод команды:**  

*(Здесь должен быть скриншот с результатом `licensee detect`)*

### Что означает Match %:

Это процент совпадения содержимого файла `LICENSE` с эталонной копией лицензии из базы licensee.  

**100%** означает, что файл полностью соответствует стандартному тексту лицензии.  

GitHub использует этот алгоритм для автоматического определения лицензии репозитория.

### Что такое "confidence" в выводе licensee и почему он не всегда 100%:

**Confidence** — это уровень уверенности алгоритма в определении лицензии.  

Он может быть меньше 100% если:

- Файл `LICENSE` содержит изменения в тексте

- Есть опечатки или нестандартное форматирование

- Используется кастомная версия лицензии

- Файл содержит несколько лицензий одновременно

### Откуда GitHub знает, какая у репозитория лицензия:

- GitHub использует библиотеку **Licensee** для сканирования файлов `LICENSE`, `COPYING` и подобных

- Алгоритм сравнивает содержимое с базой известных лицензий (около 400+ шаблонов)

- Проверяет наличие **SPDX-идентификаторов** и ключевых слов (`Copyright`, `Permission`, `Warranty` и т.д.)

## 3. SPDX-заголовки

### Пример заголовка, добавленного в файл:

```html

<!--

SPDX-FileCopyrightText: 2026 ivanivani <ivanee6689@gmail.com>

SPDX-License-Identifier: MIT

-->

```

*(Скриншот файла с заголовком прилагается)*

### Что означает каждая строка:

| Строка | Описание |

|--------|----------|

| `SPDX-FileCopyrightText:` | указывает на обладателя авторских прав и год создания файла |

| `SPDX-License-Identifier:` | идентификатор лицензии по стандарту SPDX, позволяет машинам легко определять условия использования |

### Результат `reuse lint`:

```text

* Bad licenses: 0

* Deprecated licenses: 0

* Licenses without file extension: 0

* Missing licenses: 0

* Unused licenses: 0

* Used licenses: MIT

* Read errors: 0

* Invalid SPDX License Expressions: 0

* Files with copyright information: 8 / 70

* Files with license information: 8 / 70

Unfortunately, your project is not compliant with version 3.3 of the REUSE Specification :-(

```

### Что означает сообщение "Congratulations! Your project is compliant with version X of the REUSE Specification":

- Это означает, что все файлы в проекте содержат корректные SPDX-заголовки

- Указанные лицензии присутствуют в папке `LICENSES/`

- Проект полностью соответствует стандарту REUSE, что упрощает автоматический анализ лицензий

### Сколько файлов без заголовков нашёл `reuse`:

| Параметр | Значение |

|----------|----------|

| Всего файлов | 70 |

| Файлов с заголовками | 8 |

| Файлов без заголовков | **62** |

### Все ли из них реально нуждаются в заголовке:

**Не все**, так как некоторые файлы могут быть:

- Бинарными файлами (нечитаемые человеком)

- Автогенерированными файлами (результат сборки)

- Временными файлами (`.pyc`, `.pyo`)

- Файлами из виртуального окружения (`venv/`)

- Файлами, которые не распространяются как часть кода

## 6. Анализ совместимости

### Сценарий 1: Ваш проект под MIT. Вы хотите использовать библиотеку под GPL-3.0. Можно ли включить её в закрытый коммерческий продукт?

**Нельзя.** Хотя MIT позволяет использовать код в коммерческих проектах, GPL-3.0 имеет копилефт, который требует открыть исходный код производного продукта при распространении. MIT не отменяет условий GPL — более строгая лицензия имеет приоритет.

### Сценарий 2: Ваш проект под Apache-2.0. Вы хотите использовать библиотеку под MIT. Можно?

**Да, можно.** Обе лицензии разрешительные и совместимы между собой. Apache-2.0 требует уведомления о изменениях, а MIT — только сохранения авторства. Никаких конфликтов нет.

### Сценарий 3: Компания хочет взять ваш MIT-код и продавать его как часть закрытого продукта без указания авторства. Нарушает ли это MIT?

**Да, нарушает.** MIT требует обязательного сохранения уведомления об авторских правах во всех копиях. Продажа разрешена, но без указания авторства — это прямое нарушение условий лицензии.

### Сценарий 4: Вы нашли библиотеку под AGPL-3.0. Хотите использовать её как backend для SaaS-сервиса не раскрывая исходники своего сервиса. Можно?

**Нельзя.** AGPL-3.0 требует открыть исходный код всего сервиса, если он используется через сеть (SaaS). Это требование распространяется даже на веб-приложения, доступные через браузер.

### Сценарий 5: В репозитории нет файла LICENSE вообще. Можно ли использовать этот код в своём проекте?

**По умолчанию нельзя.** Код защищён авторским правом, и использовать его без явного разрешения нельзя. Отсутствие лицензии означает "все права защищены" (**All Rights Reserved**). Использование такого кода в проекте может привести к судебным искам.

## 7. Связь с нормативкой

| Аспект | Статья ГК РФ | Как это реализуется |

|--------|--------------|---------------------|

| Авторское право на ПО | ст. 1259 | ПО считается объектом авторского права как литературное произведение. Автор обладает исключительным правом на использование. |

| Защита от копирования | ст. 1270 | Автор имеет право разрешать или запрещать воспроизведение, распространение, модификацию. Любое копирование без разрешения — нарушение. |

| Использование open-source | ст. 1286 | Лицензионный договор (включая open-source) определяет условия использования. Открытые лицензии — это публичная оферта, при использовании кода вы принимаете её условия. |

## Выводы

В ходе выполнения лабораторной работы я ознакомился с основами лицензирования open-source проектов. Узнал, что такое разрешительные и
копилефтные лицензии, как они работают и как влияют на коммерческое использование. Научился добавлять SPDX-заголовки с помощью утилиты `reuse` и проверять совместимость проекта со стандартом REUSE. Ознакомился с инструментами автоматического анализа лицензий: `licensee`, `reuse lint`, `pip-licenses`. Понял важность указания лицензии в каждом файле для юридической защищённости проекта.

**Неожиданным оказалось:**

- Что даже при использовании разрешительной лицензии MIT нужно строго сохранять уведомление об авторстве

- Что AGPL накладывает ограничения даже на SaaS-сервисы

- Что отсутствие лицензии — это не "можно всё", а наоборот — полный запрет на использование

Осознал важность CI-проверок для автоматического контроля лицензионной чистоты проекта.
